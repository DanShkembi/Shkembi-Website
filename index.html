Data • Reliability • Reproducibility

ARR Analytics Pipeline — Keeper Security

I rebuilt a spreadsheet/Tableau-based ARR calculation into a reproducible Python + SQL pipeline with monthly granularity. The goal wasn’t only speed—it was trust. I made each transformation small and testable, documented assumptions inline, and produced outputs that analysts could audit and regenerate consistently.

  • Contract windowing: allocate revenue only within contract_start → contract_end months
  • Monthly scaffold: cross-join subscriptions with a canonical “Month Start” calendar
  • ARR formula (simplified): (net_revenue - refund) / term_length_months per active month
  • Pivot to wide matrices for cohort views; lag/lead checks to detect month-level anomalies
Deep dive: approach

The pipeline reads raw subscription CSVs, normalizes dates, computes a term length in months, and derives ARR as (net_revenue - refund) / term_length. It builds a monthly calendar (“Month Start”), omits excluded months, and cross-joins the calendar to subscriptions to generate a full grid. ARR is assigned only when the calendar month falls inside the contract window.

After grouping by subscription and month, it pivots to a wide format for reporting. To catch spiky artifacts, the code aligns each row with previous/next month (lag/lead) and conditionally smooths anomalies so downstream dashboards reflect true patterns—not data hiccups.

What I learned: ARR work lives at the intersection of math and governance. Tests, lineage, and clear contracts matter as much as raw optimization.
